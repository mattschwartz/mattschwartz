Never Satisfied

Warning: This Is A Long Post...

Posted on November 17, 2015

Never Satisfied

I prototyped an interface next that I would end up using for over a year. It had everything a basic roguelike needed: an inventory, a character sheet and a world log history to keep track of the events taking place around the player.

The earliest real interface for StoneQuest

The red @ represents the player character, the purple # represents walkable tiles (note the player has moved far out of bounds in this screenshot...), green | and - represent walls and the green % was a profoundly simple enemy character. The text box at the bottom left reports events happening around the player in the world, the box to the right of that is the player's character sheet and the taller box on the far right is the player's inventory.

For over a month, I doodled in notebooks and pseudocoded algorithms to randomly generate my levels, enemies and items. I spent so much more time on the fundamental game design than actually designing the game during this period. When I decided to start implementing some things, I also started the project over from scratch. Dozens of hours later, I found myself with this:

The earliest fully-functional interface for StoneQuest

I still find it simple and elegant. It served its purpose well and was very easy to create. Near the end of this iteration, I even had color themes that the player could change at will. But I wanted more still from StoneQuest.

Graphics

So, I replaced the ASCII tile system with simple artwork that I drew.

StoneQuest starts using Canvas and a double buffer rasterizer

It is about this time when I decided on the name for my game, StoneQuest. I also added a nifty minimap in the top right. I found it necessary to put more effort on the story before I could continue with the art style, skills, setting, etc. (Read more about the story here.)

Still Not Satisfied

It still was not right. I felt the UI no longer matched the introduction of graphics.

I liked this one a lot. Still like it, in fact. But it was not very user-friendly. It was annoying to use, even.

A careful eye might notice the influences from Monster Hunter 3

So I spent nearly the entire summer in 2013 drawing and detailing every window that I would want to have in the final game: inventory, character, item upgrade, a journal, item salvaging and alchemy. I spent the remainder of the summer overhauling the internal system as well. I improved the performance of the game during the following semester by significant amounts, which allowed me to create larger maps. I also increased the size of the individual tiles to expose more detail.

And now, something a bit more Diablo 3

The more I worked in the game engine that I created in Java, the more I realized I was reinventing the wheel. I wrote my own event handler and rasterizer which took a long time and still wasn't perfect.

This prompted me to investigate alternatives. I spent some time working in a Java library called Slick2D before moving on to libGDX. But it didn't feel right. It was not as enjoyable to work in as was working in the game engine I wrote from scratch (mostly).

I really enjoyed Divinity: Original Sin, too...
